# Code Commenting Rules

Последнее обновление: 2026-04-08

Этот файл фиксирует обязательные правила комментирования кода для дальнейших проходов по проекту.

## Обязательные правила

### 1. Комментарии к функциям

Для функции обязателен комментарий с описанием алгоритма, если выполняется хотя бы одно условие:

- сложность функции выше `O(1)` или `O(n)`;
- функция длиннее `15` строк кода;
- логика работы функции неочевидна без дополнительного контекста.

Что обязательно описывать в комментарии:

- назначение функции;
- алгоритм работы по шагам;
- при необходимости оценку сложности;
- важные ограничения или инварианты, если они есть.

Формат комментария:

```ts
/**
 * ...
 */
```

### 2. Комментарии к свойствам интерфейсов

Для каждого свойства каждого интерфейса обязателен комментарий в формате:

```ts
/** ... */
property: Type
```

Требование распространяется на:

- `interface`;
- `type` с объектной структурой, если он играет роль публичного контракта;
- параметры конфигурационных объектов, если они описываются отдельным интерфейсом.

### 3. Комментарии для неочевидной логики

Если логика работы участка кода неочевидна, необходимо добавлять поясняющий комментарий перед этим участком.

Когда логика считается неочевидной:

- есть скрытое бизнес-правило;
- есть временная совместимость или workaround;
- есть merge / reconciliation / optimistic update;
- есть неявная сортировка, приоритизация или синтез данных;
- код зависит от порядка выполнения или особенностей runtime.

Формат комментария:

```ts
/*
 * ...
 */
```

или, если это описание функции/метода как целого:

```ts
/**
 * ...
 */
```

### 4. Комментарии внутри классов

Обязательно давать комментарий к методам в формате:

```ts
/**
 * ...
 */
```

Также, давать комментарий к свойствам класса в формате:

```ts
/** ... */
property: Type
```

## Практика применения

- Комментарии на русском языке
- Комментарии не должны дублировать очевидный код.
- Комментарии должны объяснять `почему` и `как`, а не только `что`.
- Для простых getter/setter и тривиальных JSX-веток комментарии не нужны.
- Если есть выбор между коротким рефакторингом и длинным комментарием, сначала предпочтителен рефакторинг.
- Если сложность функции высокая, добавлять в комментарий тег `HIGH_COMPLEXITY`

## Приоритет

При дальнейших изменениях в проекте этот файл считать рабочим стандартом комментирования.
